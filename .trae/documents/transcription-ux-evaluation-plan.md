# 转录功能模块用户体验评估与优化计划

> **目标**：系统性评估转录功能模块的完整操作流程，重点关注文件选择环节的用户体验问题，产出结构化评估报告供高级软件工程师团队进行技术实现与优化迭代。

---

## 一、评估范围与目标

### 1.1 评估范围

| 模块 | 组件 | 评估重点 |
|------|------|----------|
| 前端界面 | 文件库页面 (`library/page.tsx`) | 已下载文件的可访问性与可选择性 |

1. 验证已下载文件的可访问性与可选择性

3. 分析当前文件选择机制与用户预期行为之间的差距
4. 识别影响功能可用性的关键问题点
5. 提出具体可实施的改进建议
6. 整理成结构化的用户体验评估报告

## 二、当前实现分析

### 2.1 转录功能入口分析

当前系统中，转录功能涉及以下入口：

| 入口位置 | 功能描述 | 文件路径 |
|----------|----------|----------|
| 侧边栏导航 | 「文件库」标签 | `Sidebar.tsx` |
| 文件库页面 | 展示已下载/上传的音频文件 | `library/page.tsx` |
| 内容创作页面 | 手动粘贴转录文本进行笔记生成 | `create/ContentProcessor.tsx` |
| Dashboard概览 | 统计转录时长数据 | `dashboard/page.tsx` |

### 2.2 关键发现：文件选择机制缺失

**核心问题**：当前系统**没有直接的"选择已下载文件进行转录"功能**。

当前流程：
```
用户下载音频 → 音频保存到 data/audio/ → 文件库可查看 → 
❌ 但无法直接从文件库选择文件进行转录
```

用户若要进行转录，必须：
1. 通过其他方式（如直接调用API）上传音频文件到转录接口
2. 或在内容创作页面手动粘贴已有的转录文本

### 2.3 现有转录API分析

后端转录接口 (`backend/routers/transcribe.py`)：
- **接口**：`POST /api/transcribe` — 接收 `UploadFile` 文件上传
- **支持格式**：依赖 `ffprobe` 检测，理论上支持 mp3, m4a, aac, ogg, wav, flac
- **处理方式**：同步转录（阻塞式）
- **错误处理**：500 错误统一返回，缺乏细化的错误分类

前端内容创作页面 (`create/ContentProcessor.tsx`)：
- **输入方式**：纯文本粘贴（`<textarea>`）
- **无文件上传组件**：用户无法直接上传音频文件
- **无文件选择器**：无法从已下载文件中选择

## 三、问题识别与优先级

### 3.1 P0 — 关键可用性问题（阻塞性）

| 编号 | 问题 | 影响 | 用户场景 |
|------|------|------|----------|
| P0-1 | **无文件上传入口** | 用户无法将音频文件提交给转录服务 | 用户下载播客后，无法直接转录 |
| P0-2 | **文件库与转录流程断裂** | 已下载文件无法被转录功能使用 | 用户在文件库看到音频，但无法选择转录 |
| P0-3 | **转录结果无法回流到内容创作** | 转录文本需要手动复制粘贴 | 用户完成转录后，需手动复制到笔记生成 |

### 3.2 P1 — 体验优化问题（高优先级）

| 编号 | 问题 | 影响 | 改进方向 |
|------|------|------|----------|
| P1-1 | 文件格式限制不明确 | 用户不确定哪些音频格式可上传 | 前端增加格式提示与验证 |
| P1-2 | 转录进度不可见 | 同步转录阻塞，用户不知道处理状态 | 改为异步+进度轮询 |
| P1-3 | 转录耗时预估缺失 | 用户无法预估等待时间 | 利用后端已有的 `calculate_estimate_time` |
| P1-4 | 错误提示不友好 | 统一返回"Transcription failed" | 细化错误类型（格式不支持、文件过大等） |
| P1-5 | 无文件大小限制提示 | 大文件可能导致超时或内存问题 | 前端增加文件大小检查 |

### 3.3 P2 — 增强体验问题（中优先级）

| 编号 | 问题 | 影响 | 改进方向 |
|------|------|------|----------|
| P2-1 | 无法批量选择文件 | 多文件转录效率低 | 支持批量上传/队列处理 |
| P2-2 | 转录历史不可查看 | 无法追溯之前的转录结果 | 增加转录历史记录 |
| P2-3 | 无音频预览功能 | 选择文件前无法确认内容 | 增加音频播放预览 |
| P2-4 | 转录参数不可配置 | 语言、引擎等参数固定 | 提供转录选项配置 |

## 四、用户预期行为 vs 当前实现差距

### 4.1 用户预期流程

```
1. 用户进入「文件库」
2. 看到已下载的播客音频列表
3. 选择某个音频文件 → 点击「转录」按钮
4. 系统显示转录进度/预估时间
5. 转录完成后，自动跳转到「内容创作」并带入转录文本
6. 用户基于转录文本生成笔记
```

### 4.2 当前实际流程

```
1. 用户进入「文件库」
2. 看到已下载的播客音频列表
3. ❌ 无法直接选择文件进行转录
4. 用户必须：
   a) 手动找到音频文件路径
   b) 通过API或其他工具单独调用转录
   c) 将转录结果手动复制到内容创作页面
```

### 4.3 差距分析

| 环节 | 用户预期 | 当前实现 | 差距等级 |
|------|----------|----------|----------|
| 文件选择 | 在文件库一键选择 | 无选择功能 | 🔴 严重 |
| 转录触发 | 点击转录按钮 | 无触发入口 | 🔴 严重 |
| 进度反馈 | 实时进度+预估时间 | 同步阻塞无反馈 | 🟡 中等 |
| 结果使用 | 自动带入内容创作 | 手动复制粘贴 | 🟡 中等 |
| 错误处理 | 友好提示+解决建议 | 统一500错误 | 🟡 中等 |

## 五、改进方案设计

### 5.1 方案A：文件库集成转录入口（推荐）

**设计思路**：在文件库页面为音频文件添加「转录」操作按钮，打通文件库到转录的链路。

**实施步骤**：
1. **后端**：修改转录接口，支持通过文件路径（而非仅上传）发起转录
2. **前端**：在文件库音频卡片上增加「转录」按钮
3. **前端**：创建转录进度面板组件（类似下载管理器）
4. **前端**：转录完成后自动跳转到内容创作并带入文本

**优点**：
- 利用已有文件库基础设施
- 用户流程自然（文件库 → 转录 → 创作）
- 实现成本相对较低

**缺点**：
- 需要修改后端转录接口

### 5.2 方案B：独立转录页面

**设计思路**：创建独立的「转录」页面，支持文件上传和已下载文件选择两种模式。

**实施步骤**：
1. **前端**：新增 `/dashboard/transcribe` 页面
2. **前端**：页面包含「上传文件」和「从文件库选择」两个Tab
3. **后端**：保持现有转录接口，新增通过文件ID发起转录的接口
4. **前端**：转录结果展示+一键带入内容创作

**优点**：
- 功能集中，用户体验完整
- 支持两种文件来源（上传/已下载）
- 便于扩展更多转录配置选项

**缺点**：
- 需要新增页面和路由
- 开发工作量较大

### 5.3 方案C：内容创作页面集成上传

**设计思路**：在内容创作页面的转录输入区域，增加文件上传和文件选择功能。

**实施步骤**：
1. **前端**：在 `ContentProcessor` 组件中增加文件上传区域
2. **前端**：支持拖拽上传和点击选择文件
3. **前端**：增加「从文件库选择」弹窗
4. **后端**：保持现有接口不变

**优点**：
- 用户在一个页面完成转录+创作
- 后端改动最小

**缺点**：
- 页面职责不单一（创作+转录混合）
- 文件库和创作页面耦合

## 七、预期效果

### 7.1 用户流程优化

**优化前**：
```
下载音频 → 文件库查看 → ❌ 无法转录 → 手动找文件 → 手动调用API → 手动复制结果 → 内容创作
```

**优化后**：
```
下载音频 → 文件库查看 → 点击转录 → 查看进度 → 自动进入内容创作（带转录文本） → 生成笔记
```

### 7.2 关键指标提升

| 指标 | 当前 | 目标 | 提升 |
|------|------|------|------|
| 转录功能发现率 | ~0%（无入口） | 100% | +100% |
| 文件到转录转化率 | ~0% | >60% | +60% |
| 用户操作步骤数 | 6+步 | 3步 | -50% |
| 错误处理清晰度 | 低（统一500） | 高（分类提示） | 显著 |
