## 二、问题详细清单

### 🔴 P0 - 严重问题

#### P0-001: 转录任务 100% 失败 — "转录过程中发生错误"

| 项目 | 内容 |
|------|------|
| **问题描述** | 所有转录任务（包括中文和英文）均失败，显示"转录过程中发生错误"，用户无法获得任何转录结果 |
| **严重程度** | 🔴 严重 |
| **影响范围** | 所有用户、所有音频文件 |
| **复现步骤** | 1. 进入 Library 页面<br>2. 点击任意音频文件的"转录"按钮<br>3. 等待处理<br>4. 任务状态变为"failed"，显示"转录过程中发生错误" |
| **预期行为** | 转录成功，显示"转录完成"，并可查看/使用转录文本 |
| **实际表现** | 所有任务失败，错误信息为"转录过程中发生错误"，无具体原因 |
| **根因分析** | 1. `smart_transcriber.py:154` Whisper 回退逻辑调用 `_transcribe_with_sensevoice`，但 `_get_whisper_transcriber()` 在初始化 faster-whisper 失败时抛出异常，未被捕获<br>2. `_do_transcription` 中 `detect_language_only` 调用可能触发模型加载，导致异常<br>3. 错误分类逻辑将具体异常归类为 generic "transcription_error"，丢失原始错误信息 |
| **修复建议** | 1. 在 `_get_whisper_transcriber()` 中捕获所有初始化异常<br>2. 在 `_do_transcription` 中延迟加载模型，避免在任务启动时失败<br>3. 增强错误日志，保留原始异常堆栈 |
| **相关代码** | `core/smart_transcriber.py:154-165`, `backend/routers/transcribe.py:313-436` |

#### P0-002: 错误信息过于笼统，无法指导用户解决问题

| 项目 | 内容 |
|------|------|
| **问题描述** | 转录失败后只显示"转录过程中发生错误"，用户不知道具体原因和解决方法 |
| **严重程度** | 🔴 严重 |
| **影响范围** | 所有失败任务 |
| **复现步骤** | 1. 触发任意转录失败<br>2. 查看错误提示 |
| **预期行为** | 显示具体错误原因（如"模型未安装"、"内存不足"等）和可操作的解决方案 |
| **实际表现** | 仅显示"转录过程中发生错误"，解决方案为"如果问题持续存在，请联系技术支持" |
| **根因分析** | 1. `_categorize_error` 函数匹配逻辑过于简单，无法捕获模型加载失败等深层错误<br>2. 前端 `getErrorMessage` 对 `transcription_error` 类别返回固定文本<br>3. 原始异常信息丢失，未传递到前端 |
| **修复建议** | 1. 扩展错误分类，增加"模型加载失败"、"音频解码失败"等类别<br>2. 在 `_do_transcription` 中捕获并记录详细异常堆栈<br>3. 前端显示更具体的错误信息和解决方案 |
| **相关代码** | `backend/routers/transcribe.py:53-87`, `web-dashboard/src/app/dashboard/library/page.tsx:97-118` |

#### P0-003: 进度条卡在 20% 后长时间无更新

| 项目 | 内容 |
|------|------|
| **问题描述** | 转录进度在 20% 后长时间不更新，用户以为任务卡死 |
| **严重程度** | 🔴 严重 |
| **影响范围** | 所有进行中任务 |
| **复现步骤** | 1. 开始转录<br>2. 观察进度条到 20%<br>3. 等待数分钟，进度无变化 |
| **预期行为** | 进度条应平滑更新，每 5 秒刷新一次 |
| **实际表现** | 进度在 10% → 20% 后停止更新，直到任务完成或失败才跳到 100% |
| **根因分析** | 1. `_do_transcription` 中进度更新点只有 3 个（10%, 20%, 100%）<br>2. 进度线程 `update_progress` 依赖 `duration` 和 `estimate`，如果 `get_audio_duration` 返回 None 则不会启动<br>3. 对于长音频，20% 到 100% 之间没有任何反馈 |
| **修复建议** | 1. 确保进度线程始终启动，即使无法获取音频时长<br>2. 增加基于时间的进度估算（如每分钟增加 5%）<br>3. 添加处理阶段指示（"加载模型..."、"识别中..."） |
| **相关代码** | `backend/routers/transcribe.py:365-381` |

### 🟡 P1 - 中等问题

#### P1-001: 预估时间不准确

| 项目 | 内容 |
|------|------|
| **问题描述** | 转录预估时间与实际时间差距大，用户无法合理安排等待时间 |
| **严重程度** | 🟡 中等 |
| **影响范围** | 所有任务 |
| **复现步骤** | 1. 开始转录<br>2. 查看预估时间<br>3. 对比实际完成时间 |
| **预期行为** | 预估时间误差在 ±20% 以内 |
| **实际表现** | 预估 1 小时，实际可能 3-5 小时（CPU 环境） |
| **根因分析** | 1. `calculate_estimate_time` 使用固定速度因子，未考虑 CPU/GPU 差异<br>2. 未根据历史数据校准预估<br>3. 对于大文件，模型加载时间未计入 |
| **修复建议** | 1. 添加硬件检测（CPU/GPU）<br>2. 根据硬件动态调整速度因子<br>3. 收集历史转录数据，建立自适应预估模型 |
| **相关代码** | `backend/routers/transcribe.py:166-258` |

#### P1-002: 缺少转录控制功能（暂停/停止）

| 项目 | 内容 |
|------|------|
| **问题描述** | 转录开始后无法暂停或停止，用户只能等待完成或刷新页面 |
| **严重程度** | 🟡 中等 |
| **影响范围** | 所有进行中任务 |
| **复现步骤** | 1. 开始转录<br>2. 尝试找到暂停/停止按钮 |
| **预期行为** | 提供暂停和停止按钮，允许用户控制任务 |
| **实际表现** | 无任何控制按钮，任务一旦开始不可中断 |
| **根因分析** | 1. 前端 UI 未设计控制按钮<br>2. 后端 API 缺少暂停/停止端点<br>3. 线程池任务不可取消 |
| **修复建议** | 1. 添加暂停/停止按钮到前端<br>2. 实现 `POST /api/transcribe/{task_id}/stop` 端点<br>3. 使用 `threading.Event` 实现协作式取消 |
| **相关代码** | `web-dashboard/src/app/dashboard/library/page.tsx`, `backend/routers/transcribe.py` |

#### P1-005: 转录历史无持久化

| 项目 | 内容 |
|------|------|
| **问题描述** | 转录任务状态存储在内存中，页面刷新后丢失 |
| **严重程度** | 🟡 中等 |
| **影响范围** | 所有用户 |
| **复现步骤** | 1. 开始转录<br>2. 刷新页面<br>3. 转录状态丢失 |
| **预期行为** | 转录状态持久化，刷新后可恢复 |
| **实际表现** | 页面刷新后，进行中的任务消失 |
| **根因分析** | 1. 任务状态存储在内存 `jobs` 字典<br>2. 前端使用 localStorage 但只存储任务 ID，不存储状态 |
| **修复建议** | 1. 使用 Redis/数据库持久化任务状态<br>2. 前端恢复时查询后端状态<br>3. 添加任务历史页面 |
| **相关代码** | `backend/routers/transcribe.py:32`, `web-dashboard/src/lib/transcriptionStorage.ts` |
